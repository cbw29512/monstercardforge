# DM Forge Whole-Site Audit — 2026-07-21

## Scope

Audited repositories and modules:

- `cbw29512/monstercardforge`
  - DM Forge homepage
  - Monster Card Forge
  - Magic Item Forge
  - Session Console
- `cbw29512/healingbox`
  - Cleric in a Box

This was a source-code and architecture audit. The execution environment could not resolve the public GitHub Pages domain, so live-browser Lighthouse, axe, touch-device, cross-browser, and physical printer tests remain the first validation step tomorrow.

## Executive Assessment

DM Forge has a strong product direction and several genuinely useful modules. It is best described as a promising functional alpha rather than a production-ready unified platform.

### Strongest qualities

- Clear print-first and table-first product philosophy
- Distinct tools with understandable purposes
- Static hosting with fast startup and no mandatory accounts
- Local persistence and JSON backup in major modules
- Secure random-number generation in the dice tools
- Player HP responsibility is correctly separated from the DM tools where requested
- Good foundation for phones, tablets, laptops, and home printers

### Biggest current weakness

The modules operate as separate applications rather than one campaign system. Campaign names, data, navigation, storage, rules metadata, and exports are duplicated instead of shared.

# Critical Findings

## C-1 — Monster Card Forge renders user text as raw HTML

**Area:** Monster Card Forge homebrew rendering  
**Risk:** Cross-site scripting and loss/exposure of locally saved DM data  
**Priority:** First fix tomorrow

Monster fields such as name, type, attacks, traits, senses, source, and spell text are inserted directly into HTML strings. Homebrew form values flow into those renderers without HTML escaping.

Because Monster Card Forge, Magic Item Forge, and Session Console share the same GitHub Pages origin, malicious or accidental HTML entered into Monster Card Forge could run JavaScript that reads or modifies the other modules' `localStorage`, including private item secrets and campaign notes.

### Required fix

- Introduce one shared `escapeHtml()` utility.
- Escape every user-controlled scalar before inserting it into HTML.
- Render long-form text with `textContent` or DOM creation where practical.
- Add automated hostile-input tests using strings such as `<img src=x onerror=...>`.
- Add a Content Security Policy after inline scripts/styles have been removed.

## C-2 — Cleric in a Box identifies the DM by the display name `DM`

**Area:** Cleric in a Box multiplayer permissions  
**Risk:** A player can submit the name `DM` and request DM-only reset or undo actions  
**Priority:** First fix tomorrow

Remote actions include a user-supplied display name. DM-only actions are permitted when the actor string equals `DM`. This is not authentication.

### Required fix

- Bind permissions to the host connection, not a display name.
- Never accept an actor role from a player payload.
- Store a host-only capability token in the hosting browser.
- Reject `rest`, `undo`, settings, and administrative actions from every remote connection.
- Optionally add a room password or unguessable invitation secret.
- Add tests proving a player named `DM` cannot run host actions.

## C-3 — Magic Item cards can silently truncate rules text

**Area:** Magic Item Forge previews and print layouts  
**Risk:** Printed cards can omit rules, charges, curses, or other important content  
**Priority:** First fix tomorrow

Magic Item cards use fixed heights and `overflow: hidden`. Complex items and artifacts can exceed the available area without a warning.

### Required fix

- Detect overflow before printing.
- Show a clear `Content does not fit` warning.
- Automatically move detailed content to the back or additional pages.
- Allow artifact booklets to expand instead of clipping.
- Never reduce text below an agreed readable minimum.
- Add test fixtures containing deliberately long item text.

# High-Priority Findings

## H-1 — No shared campaign data model

Each module uses its own local-storage key and its own idea of a campaign. A campaign created in Session Console is not automatically available in Magic Item Forge or Cleric in a Box.

### Recommendation

Create a versioned shared data layer:

```text
DMForgeStore
├── schemaVersion
├── campaigns[]
├── activeCampaignId
├── monsters[]
├── magicItems[]
├── sessions[]
├── encounters[]
├── npcs[]
├── loot[]
└── settings
```

Keep large artwork blobs in IndexedDB rather than localStorage.

## H-2 — No cross-tool handoff

Current modules are linked from a homepage, but data does not flow between them.

### Needed flows

- Monster Card Forge → Encounter Forge → Session Console
- Loot Forge → Magic Item Forge
- Magic Item Forge → Session Console handouts/rewards
- Session Console → archived campaign history
- Cleric in a Box → selected DM Forge campaign

## H-3 — Rules data requires an authoritative audit

Cleric spell lists, spell formulas, slot progressions, monster data, and 2014/2024 distinctions need systematic verification. Cleric in a Box also infers a Wisdom modifier from party level, which is a homebrew assumption rather than an explicit artifact setting.

### Recommendation

- Separate 2014, 2024, edition-neutral, and homebrew data files.
- Record source/license/version on every rules record.
- Make artifact ability modifier, spell save DC, and attack bonus explicit DM settings.
- Add tests for every supported spell that produces damage or healing.

## H-4 — Peer-to-peer reliability is not guaranteed

Cleric in a Box depends on an external PeerJS CDN and PeerJS signaling. Some networks, NAT configurations, browser backgrounding, or CDN outages may break a room.

### Recommendation

- Add reconnect status and retry behavior.
- Test separate Wi-Fi and cellular clients.
- Evaluate a small hosted state service or configured TURN fallback.
- Pin and self-host the PeerJS client instead of relying exclusively on a public CDN.

## H-5 — No automated test or validation pipeline

The repository is static and currently has no package manifest or automated test suite.

### Recommendation

Add:

- `package.json`
- ESLint
- Prettier
- Vitest for pure data/rendering logic
- Playwright for browser workflows and print screenshots
- axe accessibility checks
- GitHub Actions on every push
- Link and HTML validation

## H-6 — Storage is fragile

Local storage is convenient but has limited capacity and no transactional protection. Base64 item artwork can exhaust browser storage.

### Recommendation

- IndexedDB for artwork and larger campaign data
- Storage quota meter
- Automatic backup reminders
- Versioned migrations
- Import validation and recovery mode
- Visible last-backup timestamp

## H-7 — Print layouts need real printer calibration

CSS dimensions are promising, but source inspection cannot confirm duplex alignment, printer margins, browser headers, cardstock scaling, or content fit.

### Test matrix

- Chrome and Edge
- Letter paper and cardstock
- 100% scale
- Single-sided and long-edge duplex
- Ink-saving mode
- 2.5×3.5 cards
- 5×7 artifact cards
- Fold-over monster cards
- Boss folios
- Session packets

## H-8 — Legal and licensing records are not yet complete

The site includes general independent-tool and open-license language, but each included rules record needs traceable source and license information before public distribution or monetization.

### Recommendation

- Create a full attribution page.
- Store `ruleset`, `source`, `license`, and `contentVersion` per record.
- Keep homebrew and licensed data visibly distinct.
- Avoid implying official endorsement.

# Medium-Priority Findings

## M-1 — Accessibility semantics are incomplete

- Tab controls lack full tablist/tab/tabpanel ARIA behavior.
- Current states are often visual only.
- Focus styles are not consistently defined.
- Toasts are not announced through an ARIA live region.
- Emoji icons are not consistently hidden from screen readers or labeled.
- Modal and keyboard behavior needs testing.

## M-2 — Mobile layouts need live-device testing

CSS breakpoints exist, but the following need physical testing:

- Horizontal tab scrolling
- Initiative combatant density
- Sticky preview behavior
- Dialog keyboard overlap
- Long artifact cards
- Touch targets and accidental taps
- iOS browser backgrounding for Cleric in a Box

## M-3 — Homepage and module navigation are inconsistent

The homepage is visually distinct from Monster Card Forge, and modules use different headers and navigation patterns.

### Recommendation

Create a shared header component with:

- DM Forge home
- Current campaign
- Tools menu
- Backup status
- Ruleset indicator
- Help

## M-4 — Browser prompts and confirms weaken the interface

Several workflows use `prompt`, `alert`, and `confirm`. These are hard to style, limited on phones, and disruptive to keyboard/screen-reader users.

Replace them with accessible in-page dialogs.

## M-5 — Generator content will become repetitive

Session Console generators use small embedded arrays. This is fine for a prototype but repetition will become obvious.

Expand through structured combinations and campaign-aware context rather than simply adding very large random lists.

## M-6 — Monster homebrew editing is incomplete

The homebrew builder primarily edits scalar fields. Actions, traits, reactions, legendary actions, and spells need structured add/edit/reorder/remove controls and saving/export.

## M-7 — Session Console roster management is indirect

Players are added to a roster by entering them as combatants. Add a dedicated party profile with character name, initiative modifier, passive perception, AC, and optional notes—while continuing not to track player HP.

## M-8 — Session archive restore can overwrite current work

Restore correctly warns, but there is no automatic snapshot before replacement.

Create an automatic recovery snapshot before destructive operations.

## M-9 — No offline/PWA support

The static tools are good candidates for installation and offline use, except multiplayer signaling. Add a web manifest, icons, and service worker for core assets.

## M-10 — No centralized error reporting

Errors mostly go to the console or alerts. Add user-friendly error states and an optional downloadable diagnostic report.

# Low-Priority Findings

## L-1 — SEO and sharing metadata

Add canonical URLs, Open Graph metadata, social preview artwork, sitemap, robots file, favicons, and structured software-application data.

## L-2 — Performance consistency

- Self-host or consolidate fonts.
- Avoid repeating large inline styles.
- Add cache-friendly shared assets.
- Lazy-load nonessential modules.

## L-3 — Empty and first-run experiences

Add small guided examples and a dismissible first-run checklist for each tool.

## L-4 — Naming consistency

Standardize `DM Forge`, module titles, ruleset labels, campaign naming, and item rarity language across every module.

## L-5 — Undo and recovery consistency

Magic Item Forge, Session Console, and Cleric in a Box use different undo/recovery patterns. Define one common approach.

## L-6 — Search and filtering

As libraries grow, add full-text search, tags, favorites, recently used, and campaign filters across all tools.

## L-7 — Documentation

Add short user guides, print guides, data backup instructions, privacy explanation, and troubleshooting for multiplayer rooms.

## Definition of Done for the Hardening Pass

- No user-entered text can execute as HTML or JavaScript.
- Remote players cannot perform host-only actions.
- No printable content disappears without a warning or additional page.
- Critical rules calculations have automated tests.
- Every module passes keyboard and basic screen-reader checks.
- Every live route has a browser smoke test.
- Campaign data has a versioned backup and migration path.
- The main card and packet formats pass physical print tests.

# Audit Conclusion

Do not add another major module before completing the safety, correctness, testing, and shared-campaign foundation above. The right next move is to make the existing tools trustworthy and connected. After that, Encounter Forge will add far more value because it can hand real monster and encounter data directly into a stable Session Console.
