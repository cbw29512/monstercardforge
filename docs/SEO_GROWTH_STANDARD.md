# DM Forge SEO and Growth Standard

**Version:** 1.0.0  
**Effective date:** 2026-07-21

DM Forge should grow because it solves real Dungeon Master problems exceptionally well—not because it imitates another brand, publishes thin keyword pages, or manipulates ranking systems.

## Growth principles

1. **People first.** Every indexable page must satisfy a real DM task or answer a real question.
2. **Original utility.** Interactive tools, printable outputs, documented testing, and practical examples are the primary competitive advantages.
3. **Clear trust signals.** Rules sources, verification dates, licensing, change history, privacy boundaries, and author/project information must be visible.
4. **Fast and stable.** Performance is part of the product and part of discovery.
5. **Crawlable structure.** Important tools and guidance pages must be linked with ordinary HTML anchors.
6. **No search spam.** Do not create mass-produced pages whose primary purpose is ranking rather than helping.

## Technical SEO contract

Every public indexable page must have:

- A unique, concise `<title>`
- A unique meta description
- One clear H1 matching the page purpose
- A canonical URL
- Crawlable internal links
- A mobile viewport declaration
- Meaningful HTML landmarks
- Descriptive link and button text
- A valid status code when deployed

The site root must provide:

- `robots.txt`
- `sitemap.xml`
- `WebApplication` or appropriate structured data
- Open Graph metadata
- A stable canonical hostname

## Content architecture

### Product pages

Each major tool should eventually have a public explanation section covering:

- What problem it solves
- Who it is for
- How it works
- What stays private
- What rulesets it supports
- How to print, export, or share results
- How the tool was tested
- A direct link to the tool

### Search-intent guides

Create guides only when DM Forge can add original value. Strong candidates include:

- How to build a balanced 5e encounter
- How 5e and 5.5e encounter budgets differ
- How to print double-sided D&D item cards
- How to run initiative without tracking player HP
- How to create player-safe NPC handouts
- How to prepare a reusable session worksheet
- How to back up browser-based campaign data

Every guide must include first-hand screenshots, examples generated by DM Forge, links to the relevant tool, and source attribution where rules are discussed.

### Pages to avoid

- Thin pages for every spell or monster without original utility
- Automatically generated synonym pages
- Copied rulebook text
- Unverified “best” lists written only for traffic
- Pages targeting unrelated tabletop or gaming topics
- Doorway pages that lead to the same tool

## Search-result presentation

- Titles should state the task and product clearly.
- Avoid exaggerated words such as “perfect,” “guaranteed,” or “official” unless literally accurate.
- Meta descriptions should explain the result a visitor receives.
- Major pages should receive a purpose-built share image before social promotion begins.
- Structured data must describe visible content and must not invent ratings, reviews, or organizations.

## Performance targets

Measure field data at the 75th percentile, separated for mobile and desktop:

- Largest Contentful Paint: **2.5 seconds or less**
- Interaction to Next Paint: **200 milliseconds or less**
- Cumulative Layout Shift: **0.1 or less**

Implementation priorities:

1. Eliminate unnecessary remote font families.
2. Avoid large uncompressed artwork in initial page loads.
3. Reserve dimensions for images and asynchronous content.
4. Keep JavaScript modules small and page-specific.
5. Cache static assets when PWA support is introduced.
6. Measure live pages rather than relying only on laboratory tests.

## Trust and expertise

Public guidance pages should state:

- Who created or maintains the project
- Why the tool exists
- How rules were verified
- Which sources and licenses apply
- When the page was last reviewed
- Whether content is official rules, open-license material, original DM Forge content, or homebrew

AI-assisted work must still be reviewed, sourced, tested, and owned by the project maintainer. Automation is not a substitute for verification.

## Traffic channels

Priority order:

1. Search traffic from useful guides and tools
2. Direct sharing between DMs
3. GitHub discovery and open development history
4. Relevant tabletop communities where self-promotion is allowed
5. Demonstration videos showing real table workflows
6. Printable samples that contain a subtle DM Forge URL or QR code

Do not spam communities, automate unsolicited promotion, or disguise marketing as neutral advice.

## Measurement plan

Track only privacy-respecting metrics needed to improve the product:

- Tool launches
- Successful exports and prints
- Cross-tool handoffs
- Return visits
- Guide-to-tool conversion
- Mobile versus desktop usage
- Core Web Vitals
- Search queries and landing pages through Google Search Console

Analytics must be documented before introduction. Do not collect campaign content, room history, NPC secrets, or player names.

## Release checklist for indexable pages

- [ ] Unique title and description
- [ ] Canonical URL
- [ ] One descriptive H1
- [ ] Crawlable internal links
- [ ] Valid structured data where appropriate
- [ ] Mobile and keyboard test
- [ ] Rules/source review when applicable
- [ ] No private campaign data in rendered source
- [ ] Added to sitemap
- [ ] Added to automated metadata checks
- [ ] Reviewed for useful, original value
